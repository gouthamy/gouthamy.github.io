01
HTC Global Services · 2012–2016

The Problem 20 Teams Couldn't Crack in 5 Years

The situation

HTC had a fleet of 150+ desktop applications built on a proprietary licensed image processing framework. The annual licence cost was enormous — over $1M. For five years, team after team had tried to migrate to open-source. All 20+ attempts failed. The project was considered technically impossible internally.

What I did differently

I approached it as an architecture problem, not a migration problem. Instead of trying to swap one framework for another, I designed a new abstraction layer using JavaFX — building a custom rendering engine that replicated the proprietary API surface while running entirely on open-source foundations. I built and shipped all 150+ applications on the new platform alone.

The lesson: When everyone fails at the same approach, the problem isn't the execution — it's the framing. I reframed the migration as a platform architecture challenge, which made the solution obvious.

02
PayPal via DXC Technology · 2016–2017

From Monolith to 20+ Microservices: Designing PayPal's Regulatory Platform from Scratch

The situation

PayPal's Regulatory Platform Technology (RPT) was a monolithic Java application responsible for scanning millions of customer profiles for financial crime — money laundering, terrorist financing, sanctions violations. It was slow, brittle, and couldn't scale to the velocity PayPal needed.

What I designed and built

I designed the entire microservices architecture for RPT from the ground up — decomposing the monolith into 20+ independent services, each owning a bounded context of the regulatory domain. Key technical choices I made: Apache Lucene + Levenshtein edit distance for fuzzy name matching against sanctions watchlists (tolerating transliteration errors in international names), batch-indexed world-check file processing to reduce scan time complexity from O(n²) to near-linear, and a Node.js web interface for regulatory body access to high-risk transaction data.

20+Microservices designed solo
MillionsCustomer profiles scanned
O(n²)→O(n)Scan complexity reduction
💡

The lesson: Fuzzy matching in financial compliance isn't a nice-to-have — "Osama bin Laden" and "Usama bin Ladin" are the same person. Choosing the right algorithm (edit distance) over the obvious one (exact match) was the architectural decision that made the whole system viable.

03
Amazon Advertising · 2019–2025

Architecting the Pipeline That Powers Amazon's Ad Rendering at 1M TPS

The situation

Amazon's Sponsored Display team needed a rendering service that could handle the full complexity of ad personalization — dynamic product data, real-time bidding signals, device targeting, and creative optimization — at the scale of Amazon's global retail traffic. The existing system couldn't handle the throughput or cost demands.

What I architected

I was one of the core engineers who designed the end-to-end ad rendering pipeline — from customer interaction signals flowing in through Kinesis, through the real-time bidding and targeting layer, to the rendering service that assembled and served the final ad creative. Key architectural decisions: a distributed in-memory cache for Amazon product data (eliminating synchronous catalog calls on the hot path), re-architecture to native AWS (EC2, Lambda, Kinesis), and a self-debugging React operations tool that gave dependency teams visibility into the pipeline without engineering escalations.

1M TPSPeak throughput achieved
<20msp99 latency
$8.7MCombined revenue + savings
💡

The lesson: At 1M TPS, every synchronous call is a liability. I moved Amazon product data access off the critical path entirely with a distributed cache layer — turning a potential 50ms catalog lookup into a sub-millisecond memory read. Architecture is about eliminating latency before it exists.

Engineering Philosophy

How I think about building

Principles forged across 14 years of high-stakes systems. Not slogans — things I've had to actually live.

Reframe before you build

When a problem resists 20 attempts, the answer isn't to try again — it's to question the frame. At HTC, I didn't migrate legacy software. I designed a new platform abstraction. The solution was invisible until I stopped seeing it as a migration problem.

Eliminate latency before it exists

At 1M TPS, the most expensive operations are the ones on the critical path. I design systems to move expensive work off the hot path — caching, pre-computation, async pipelines — before performance becomes a problem, not after.

Algorithm choice is architecture

At PayPal, choosing Levenshtein distance over exact string matching wasn't a coding detail — it was an architectural decision that determined whether the entire system was legally viable. The right algorithm is as important as the right service boundary.

Ownership means the whole system

I don't hand off. When I designed PayPal's regulatory microservices, I owned the domain model, the API contracts, the data pipeline, and the operational tooling. When you own the whole system, you make architectural decisions no single-component owner can make.

Build for the operator, not just the user

The self-debugging React ops tool I built at Amazon saved 10 hours/week for every dependency team. The most underbuilt part of any system is the tooling that helps engineers understand and operate it. Observability is a first-class feature.

Cognitive architecture is the next frontier

My research on adaptive cognitive architectures argues that enterprise systems are evolving from reactive to intelligent — capable of contextual interpretation and autonomous decision-making. I'm not just watching that transition. I'm building the infrastructure for it at Salesforce.
